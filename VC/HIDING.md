# Hiding: footgun drill

Five scenarios motivating the two-typeclass split between `VectorCommitment` and `HidingVectorCommitment` in `VC`. Compares the single-typeclass design (one class, hiding encoded via `Salt = Unit`) to the two-typeclass design (chosen). Mirrors the Rust analysis at [`ark-mt/HIDING.md`](https://github.com/arkworks-rs/ark-vc/blob/main/crates/ark-mt/HIDING.md); see [DESIGN.md §3.1](DESIGN.md) for the `MerkleHasher.Salt` associated-type machinery this drill rests on.

The two-typeclass design fixes Footguns 1, 4, and 5 structurally; Footguns 2 and 3 are fixed orthogonally and hold under either architecture.

**Lean-specific note on zero-cost `Salt = Unit`.** Just as Rust's `()` is a ZST that compiles away, Lean's `Unit` is reduced by the kernel: `Unit`-valued fields in a structure are erased during compilation, and `sampleSalt h seed = ()` becomes a constant. The non-hiding code path therefore pays no runtime cost for the salt machinery — the difference between the two designs is purely about *what the type system forces callers to acknowledge*, not about generated code.

## Footgun 1: ZK-required callsite accepts a non-hiding commitment

**Situation.** A SNARK prover's witness-commit function is written as
```lean
def commitWitness {V : Type*} [VectorCommitment V] (v : V) (w : Witness) : ... := ...
```
A caller hands in a concrete type whose hasher has `Salt = Unit`. Soundness still holds, but witness symbols leak to any adversary with the commitment. The code compiles and passes all correctness tests — the bug is invisible at the type level.

- **Single-typeclass design**: not caught. The author *could* add an extra `[HidingSalt (Hasher V).Salt]` instance bound, but nothing forces them to. Documentation-driven discipline.
- **Two-typeclass design (chosen)**: the author writes `[HidingVectorCommitment V]`. Non-hiding concrete types fail to elaborate at that bound — Lean's instance search refuses to find a `HidingVectorCommitment` instance for a `MerkleCommitment H S` whose `H.Salt = Unit`.

## Footgun 2: Salt reuse across commits

**Situation.** A developer writes `commit msg fixedSalt` for deterministic unit tests, then forgets to switch to random salts in production. Hiding silently broken.

- **Both designs**: already fixed. The API takes a seed (`ULift UInt64`) routed through `MerkleHasher.sampleSalt`, not a salt directly. `sampleSalt` is the only way to produce a `Salt`, and it's owned by the scheme. Callers cannot hand in a specific salt — they can only pick the seed source.

## Footgun 3: Hiding proof verified with a non-hiding verifier (or vice versa)

**Situation.** A proof generated by the hiding scheme is fed to the non-hiding verifier, or vice versa. Silent success or silent failure.

- **Both designs**: fixed. `OpeningProof H` is parametric in the hasher `H`, and `H.Salt` differs between hiding and non-hiding hashers. Lean's elaborator treats `OpeningProof Blake3Hasher` and `OpeningProof Blake3HidingHasher` as distinct types — `check` on the wrong pair simply fails to typecheck.

## Footgun 4: Non-hiding preset imported by a ZK-required caller

**Situation.** A newcomer grabs a preset like `Blake3.binary` (non-hiding) from [`ark-mt/src/presets.rs`](https://github.com/arkworks-rs/ark-vc/blob/main/crates/ark-mt/src/presets.rs) (or its Lean mirror in [`Src/Merkle/Instance.lean`](Src/Merkle/Instance.lean)) because it's the obvious default. They don't know whether it's hiding or not; the name doesn't signal it. Their downstream code treats it as hiding.

- **Single-typeclass design**: relies on naming discipline (`Blake3.binaryNonhiding` vs `Blake3.binaryHiding`). A newcomer who reads the docs carefully is fine; one who skims is not.
- **Two-typeclass design (chosen)**: same naming discipline at the preset level — but if the downstream function bounds on `[HidingVectorCommitment V]`, the wrong preset fails to elaborate at the call site. The mistake is caught *downstream* even if the preset-picker doesn't notice the name.

## Footgun 5: Impl-side degenerate-case path

**Situation.** In a single-typeclass design, every `instance : VectorCommitment _` body has to handle `Salt = Unit` alongside the hiding case in the same code body. Even though `Unit` is zero-sized and the kernel reduces it away, reviewers still have to verify the degenerate path: salt ordering, RO absorption length, transcript layout, equivocation simulator output. Hiding-specific logic and non-hiding logic share one source file.

- **Single-typeclass design**: survives. Shared code paths have to reason about both cases, even if the kernel elides the non-hiding branches.
- **Two-typeclass design (chosen)**: hiding-specific logic lives in the `instance : HidingVectorCommitment (MerkleCommitment H S)` block in [`Src/Merkle/Instance.lean`](Src/Merkle/Instance.lean) only. The non-hiding instance body has zero salt-related code to reason about.

## Scoreboard

| Footgun | Single-typeclass | Two-typeclass (chosen) |
|---|---|---|
| 1. ZK code accepts non-hiding | unfixed | fixed |
| 2. Salt reuse | fixed by seed API | fixed |
| 3. Proof type confusion | fixed by parametricity | fixed |
| 4. Preset misuse downstream | unfixed | fixed |
| 5. Impl-side degenerate path | survives | fixed |

If you're revisiting this design: argue against each footgun. If you can eliminate all five with a different encoding (e.g. a `Hiding : Bool` field on `VectorCommitment` plus a `Decidable`-flavored bound), that's worth discussing. If you can't, the two-typeclass split in [`Src/Trait.lean`](Src/Trait.lean) stays.
